# コードレビュー結果（review-010）

リポジトリ全体を確認し、実害のあるものから順に指摘をまとめました。今回は CI ワークフローの破損、API 保護の抜け道、運用時のメモリ増加、テンプレート性能の劣化が主な論点です。

## 1. `update-coverage.yml` がクリーンな CI で失敗する

**対象**: `.github/workflows/update-coverage.yml:19-31`  
**重要度**: P0

このワークフローは `pytest` と `pytest-cov` しか入れておらず、アプリ本体の依存関係をインストールしていません。そのため、クリーンな GitHub Actions runner では `app.main` の import 時に `fastapi` などが見つからず、`coverage.xml` の生成前に落ちます。

影響:
- README のカバレッジ自動更新が動かない
- `push` 時の自動メンテナンスが機能しない
- ワークフローが成功前提で作られているのに、実際には完走できない

修正案:
- `uv sync` を入れてプロジェクト依存を先に解決する
- もしくは最低でも `fastapi` / `uvicorn` / `jinja2` / `pydantic` など、`app` の import に必要な依存を入れる
- 既存の CI と揃えるなら `uv` ベースに寄せるのが素直です

## 2. レート制限が `/api/` にしか掛かっていない

**対象**: `app/main.py:47-69`, `app/main.py:82-145`  
**重要度**: P1

`rate_limit_middleware` は `request.url.path.startswith("/api/")` の場合だけ動きます。一方で、実際に計算を実行する `/calculate` は無制限です。つまり、同じ計算処理を HTML フォーム経由で叩けば、API 側の制限を簡単に迂回できます。

影響:
- 想定した abuse 対策になっていない
- 計算コストの高い式を大量送信される経路が残る
- `/api/calculate` だけ守っても、実運用の防御としては穴が空いたまま

修正案:
- `/calculate` も同じレート制限対象に含める
- もしくは計算本体の前に共通の limiter を置く
- UI 由来の POST でも同じ制御を通す

## 3. レート制限ストアの掃除が不十分

**対象**: `app/main.py:56-64`  
**重要度**: P2

期限切れエントリの削除が「現在の client の bucket が新規開始になったとき」にしか走りません。長時間にわたって多くの異なるクライアントが出入りする環境では、古いキーが残り続けやすく、`_rate_store` が不要に膨らみます。

影響:
- 一時的なクライアントが多い環境でメモリを無駄に消費する
- 実装が小さいわりに、長期運用では地味に効くリークになる

修正案:
- 毎リクエストで期限切れエントリをスイープする
- もしくは TTL 付きの外部ストアに切り替える

## 4. Jinja テンプレートのキャッシュを明示的に無効化している

**対象**: `app/main.py:38-39`  
**重要度**: P2

`templates.env.cache = {}` によって、Jinja2 のテンプレートキャッシュが無効化されています。`Jinja2Templates` を使っている意味が薄れ、リクエストごとにテンプレートの再パースが発生します。

影響:
- 低コストな UI であっても、無駄な CPU と I/O を毎回使う
- 本番でトラフィックが増えたときに効いてくる
- 変える必要がないなら、標準キャッシュを壊さない方がよい

修正案:
- 明確な理由がないなら `templates.env.cache = {}` を外す
- 開発時の再読み込みは `uvicorn --reload` 側に任せる

## まとめ

最優先は `update-coverage.yml` の依存解決です。そこが壊れていると、README の自動更新自体が成立しません。次に、`/calculate` を含めたレート制限の統一と、`_rate_store` の掃除方法の見直しです。テンプレートキャッシュの無効化は即死ではないものの、今のコードの規模でも不要なコストになっています。
