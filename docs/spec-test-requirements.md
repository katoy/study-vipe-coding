テスト要件

目的: 品質を保ちつつ自動化により回帰を防ぐ

1. 単体テスト (Unittests)
- 対象: app.services.calculator の全ロジック、app.main のAPI処理
- カバレッジ目標: core ロジック 100%、全体 90% 以上
- 実行: pytest を用いローカル・CIで実行

2. 統合テスト
- 対象: API エンドポイント（/api/calculate）と主要なエラーパス
- 実行: pytest + httpx 実装の live_server fixture

3. E2E テスト
- 対象: ブラウザ UI 操作（Playwright）
- 実行: Docker コンテナ内で Playwright を用いて実行するスクリプトを用意
- CI: リソース制約のため別ワークフローまたは条件付きで実行

4. 静的解析
- Ruff と Mypy を CI で実行。コミット前にローカルチェックを推奨。

5. カバレッジ計測
- pytest-cov を使用。CI での測定を行い、カバレッジレポートをアーティファクトとして保存可能にする。

6. テストデータとシード
- 乱数や外部リソースに依存するテストは避ける。必要なら固定シードとモックを用いる。

7. 実行フロー
- PR 作成時: ユニットテスト + Lint + Mypy を必須で実行
- main ブランチへマージ時: 上記に加え E2E（条件付き）とカバレッジレポート生成

8. 失敗対応
- テスト失敗時は該当箇所の担当にアサインし、修正PRを作成する運用ルールをドキュメント化する。
