# コードレビュー結果（テスト・保守性編 - review-002）

テストおよび長期的な運用・保守（メンテナンス）の観点から、プロダクション環境を見据えたシステムとして不足している点や、さらに堅牢にするための改善案をまとめました。

## 1. テスト（品質保証）の観点

### ① E2E（ブラウザ操作）テストの実装漏れ
- **現状**: `pyproject.toml` の開発依存関係（dev）には `pytest-playwright` が含まれていますが、現状の `tests/` には FastAPI の `TestClient` を用いたサーバー側のユニットテスト・結合テストしかありません。
- **改善案**: HTMX を用いたフロントエンドの動的DOM更新（ボタンを押して非同期で結果が表示される挙動）が正しく機能するかを担保するため、Playwright を使った E2E（End-to-End）テストを追加すべきです。

### ② CI/CD（自動テスト・自動検証）の欠如
- **現状**: GitHub Actions や GitLab CI などの継続的インテグレーション（CI）の設定ファイルが存在しません。
- **改善案**: リポジトリに `.github/workflows/test.yml` などを追加し、コードが Push された際や Pull Request が作成された際に、自動で `uv run pytest` とカバレッジ測定が走る仕組みを構築することで、将来的なデグレ（修正による既存機能の破壊）を未然に防ぐことができます。

## 2. 保守性・運用（モニタリングとデバッグ）の観点

### ① ロギング（ログ出力）の不在と過剰に広い例外キャッチ (`app/main.py`)
- **現状**: `calculate` および `api_calculate` エンドポイントにて `except Exception:` を使用しており、すべての例外を一律でキャッチして「計算式が正しくありません」として処理しています。
- **改善案**:
  1. **ログの導入**: `logging` や `loguru` などのライブラリを使用し、エラーのスタックトレースをサーバー側の標準出力に出すようにします。現状では本番環境で何のエラーが起きたのか追跡できません。
  2. **例外の絞り込み**: `except Exception` のような広範なキャッチは、コードのタイポや未定義変数などのシステムバグまで隠蔽してしまいます。`SyntaxError` や `ValueError` など、ユーザー入力起因の例外だけをキャッチするように見直すべきです。

### ② 静的型解析（Linter / Formatter）の未設定
- **現状**: コード内に型ヒント（Type Hints）は書かれていますが、コードの品質を均一に保つための Linter や Formatter（例: `Ruff`, `Black`, `mypy`）の設定や強制ルールがありません。
- **改善案**: `pyproject.toml` に `Ruff` や `mypy` の設定を追記し、スクリプトや CI で静的型チェックとコードフォーマットを自動化することで、誰がコードを書いても一定の品質と保守性が保たれるようにします。

### ③ APIエンドポイントの CORS 設定（将来の拡張性）
- **現状**: `/api/calculate` として JSON API を提供していますが、CORS (Cross-Origin Resource Sharing) 設定が存在しません。
- **改善案**: 現在の仕様（同一ドメイン上のHTMXから利用）では問題ありませんが、将来的に外部のSPA（React等）や別ドメインからこのAPIを叩かれることを想定する場合、FastAPI の `CORSMiddleware` を導入しておく必要があります。
